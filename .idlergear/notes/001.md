**Authentication Architecture Decision: Google OAuth + Session Tokens**

We've chosen a hybrid authentication model for the Cloudflare Worker queue system:

**Core Strategy:**
1. Primary: Google OAuth (PKCE flow) for human users
2. Session tokens for daily API operations (30-day TTL)
3. Static tokens for CI/CD automation

**Key Security Properties:**
- Leverages Google's 2FA and security infrastructure
- Email allowlist for access control
- Session tokens stored in KV with expiration
- No password management burden
- Multi-device support with individual session tokens

**Implementation Phases:**
- Phase 1: OAuth login flow (`forebay login`)
- Phase 2: Session token validation in Worker
- Phase 3: Token management commands

**Why this approach:**
- Better than static tokens alone (can't rotate easily, no expiry)
- Simpler than full OAuth per-request (performance)
- More secure than API keys (2FA, breach detection)
- Familiar UX (similar to `gh auth login`, `gcloud auth login`)

**Requirements:**
- Google Cloud OAuth 2.0 Client ID
- Cloudflare KV for session storage
- Worker env vars: ALLOWED_EMAILS, GOOGLE_CLIENT_ID
- CLI must implement OAuth PKCE flow with local callback server

**Alternatives Considered:**
1. Bearer tokens only - rejected (no rotation, no expiry)
2. OAuth per-request - rejected (latency, complexity)
3. Password auth - rejected (security burden)
